# leandown

A literate Lean 4 publishing tool. Annotate your Lean source files with structured
comments and `leandown` converts them to a static website — with LaTeX math via KaTeX,
clickable cross-references between files, and live rebuilds while you work.

## Installation

```
cargo install leandown
```

Requires Rust 1.80+. Lean's `lake` must be on your PATH for build validation.

For CSS bundling, Node.js and npm are needed once per project (see [Frontend bundle](#frontend-bundle)).

## Quick start

```sh
# 1. Scaffold the frontend bundle in your Lean project
leandown init path/to/your/project

# 2. Install CSS dependencies
cd path/to/your/project/leandown_site
npm install

# 3. Build the site
leandown_site/script/build

# 4. Or start the dev server (rebuilds on .lean and CSS changes)
leandown_site/script/server
```

The generated site lands in `leandown_site/output/`.

## Annotating Lean files

Add a leandown header to any `.lean` file you want to appear on the site:

```lean
-- leandown
-- [meta]
-- title = "My Page Title"
-- group = "Section Name"
-- [content]
```

Everything **before** `-- leandown` is scanned for `import` lines (used to generate
cross-page links). Everything **after** `-- [content]` is rendered as follows:

- Lines beginning with `-- ` are **prose**, rendered as Markdown + LaTeX.
- All other lines are rendered as **Lean code blocks**.
- Blank lines separate paragraphs within prose, or terminate a code block.

### Prose and math

```lean
-- This is prose. Math works inline: $f : \{0,1\}^n \to \{0,1\}^n$
-- and displayed:
-- $$\Pr[\mathcal{A}(f(x)) = x] \leq \epsilon(n)$$
```

### Numbered labels and cross-references

Use `{{tag}}` in a prose block to insert an auto-numbered label. The declaration
immediately following is registered in the cross-reference table.

```lean
-- ## {{theorem}}: Hardness of Inversion
--
-- For all PPT adversaries $\mathcal{A}$, the advantage is negligible.
theorem hardness_of_inversion ... := by
  ...
```

This renders as "Theorem 1" with an anchor. Other pages can link to it by declaration
name using backtick syntax:

```lean
-- By `hardness_of_inversion`, we know that...
```

Or by label:

```lean
-- See [ref:Theorem 1] for the full statement.
```

### Hidden sections

Wrap boilerplate in a collapsible toggle:

```lean
-- [hidden: Helper Definitions]
-- These are standard but noisy; expand if needed.
abbrev FixedBitString (n : ℕ) := List.Vector Bool n
-- [/hidden]
```

## Frontend bundle

Running `leandown init` scaffolds `leandown_site/` with:

```
leandown_site/
  package.json       ← Tailwind v4 + Typography plugin
  src/
    input.css        ← your CSS entry point; edit to customise styles
  script/
    build            ← one-shot build (calls leandown build)
    server           ← dev server with live reload (calls leandown serve)
  .gitignore
  output/            ← generated site (created on first build)
```

After `npm install`, use the scripts from your project root:

```sh
leandown_site/script/build    # build once
leandown_site/script/server   # dev server on localhost:8000
```

These are thin wrappers around `leandown build/serve` that set the right `--root`
and `--output` paths automatically. Edit `src/input.css` to add plugins, override
styles, or change typography — it's a standard Tailwind v4 project.

The scripts are always regenerated by `leandown init` to stay in sync with the
installed binary. Customise the underlying commands by editing them directly.

If `node_modules` is absent, the binary falls back to an embedded pre-built CSS so
builds don't fail in CI.

## Commands

```
leandown init [ROOT]            Scaffold leandown_site/ (safe to re-run)
leandown build [--root ROOT]    Build the site; validates each file with `lake build`
leandown serve [--root ROOT]    Build, watch .lean files, and serve on localhost:8000
               [--output DIR]   Override the output directory
               [--port PORT]    Override the port (default: 8000)
```

## How validation works

Before rendering, each annotated file is compiled with `lake build <module>`. Files
with build errors or warnings are excluded from the site. This ensures the published
site only contains correct Lean — broken proofs and `sorry`s stay off the page.
